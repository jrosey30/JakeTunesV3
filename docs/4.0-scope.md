# JakeTunes 4.0 — Scope

**Status:** Draft, 2026-04-26
**Theme:** Feature-additive release. UI enhancement + smarter AI. **Not a rewrite.**
**Owner:** Jake

This is the source-of-truth document for 4.0. Every brief during the 4.0
cycle should reference a section here. If a feature isn't in this doc,
it's out of scope until added.

---

## 1. Headline Goals

1. **UI enhancement** — visual / interaction polish on the existing 2006-iTunes shell.
2. **Smarter recommendations** — upgrade the existing Claude-driven recs with richer signals and tighter identity-based filtering.
3. **Music Daily News** — new surface delivering fresh music journalism / release news / artist updates.
4. **Music Man v2** — upgrades to the existing DJ commentary system.
5. **Equalizer** — 2006-iTunes-style multi-band EQ with presets and custom curves.
6. **Airfoil integration** — second route alongside native AirPlay, using Rogue Amoeba's Airfoil for broader speaker reach (Bluetooth, non-AirPlay receivers, multi-room).
7. **Audio analysis enrichment** — local BPM, musical key, and Camelot-wheel detection for every track in the library. Foundation for DJ-grade transitions, harmonically-compatible playlists, and Music Man's musical literacy.
8. **Crossfade Songs** — iTunes-faithful gapless transition where one track fades out as the next fades in. New Settings modal hosts the toggle + duration slider (1–12s, default 6s, default off). Bypassed when DJ Mode is active.

Anything outside these eight lanes is 4.x or later.

## 2. Foundation Work (must land before features)

These are not user-facing but every 4.0 feature depends on them. Doing
them late means rework.

### 2.1 Signal layer expansion
**Why:** Recommendations and "for you" surfaces need per-track signals
the current `ListenerProfile` doesn't store.

**What's missing today** (from audit 2026-04-26):
- `lastPlayedAt` is not persisted per-track; only a 200-item ring buffer.
- Skip count is per-artist only, not per-track.
- No time-of-day / day-of-week pattern data.
- No playlist co-occurrence (which tracks tend to play together).

**Acceptance criteria:**
- Per-track `lastPlayedAt: number` (epoch ms), updated on play complete.
- Per-track `skipCount: number`, incremented on skip-before-30s.
- Backfilled for existing library (set to 0 / null on first run, no migration drama).
- Persisted in the same store as the rest of the Track type.

### 2.2 Identity-based filtering
**Why:** Current recommendation post-filter dedupes by lowercased / article-stripped text. CLAUDE.md flags this exact pattern as the cause of the verify-and-repair incident. A 4.0 surface that says "you might like this album" but recommends an album the user already owns (because "Pt." ≠ "Part.") is the same failure mode.

**Status (2026-04-26):** Phase 0 shipped a smarter text matcher (parens, abbreviations, diacritics, articles) at `src/main/index.ts`'s `normForOwnership`. True MBID-based identity is a separate brief: `docs/4.0-mbid-backfill.md` — needs MusicBrainz Release Group ID storage plus a backfill scan. Smart-text matcher remains as the fallback when RGID is absent.

**Acceptance criteria:**
- Album dedup uses MusicBrainz Release Group ID where available, falling back to artist+normalized-title with conservative match (no aggressive normalization). [RGID portion deferred to MBID brief.]
- Track dedup uses `audioFingerprint` if present, else stable composite key.
- Text comparison is **never** the sole gate on a destructive or user-visible filter (per CLAUDE.md).

### 2.3 Claude API cost / rate-limit posture
**Why:** Audit shows ~10–20 Claude calls per active session, no caching, no batching. 4.0 adds Daily News and richer recs on top.

**Acceptance criteria:**
- Per-session call counter visible in dev mode.
- Hard daily ceiling configurable in settings (default: TBD by Jake).
- Graceful fallback path for every Claude call (cached last response, or static copy) — no broken UI when the API is down.
- Prompt caching enabled where prompt prefix is stable (Music Man core persona, taste profile).

### 2.4 Audio analysis enrichment (BPM, key, Camelot)
**Why:** DJ workflows need beat-matched and harmonically-compatible
transitions. Music Man v2 (§6) and Auto-DJ benefit directly. Smart playlists
("warm-up set: 110–120 BPM", "all in 8A or compatible") become possible.

**Decision (2026-04-26):** Local analysis. Tunebat-scrape and Spotify
audio-features were considered and rejected — Tunebat has no documented API
and would miss every non-Spotify track in the library; Spotify's
audio-features endpoint is closed to new app registrations as of late 2024.
Local analysis is offline, has no rate limits, covers every file regardless
of streaming-service availability, and is one-time CPU cost on import.

**Tools:**
- **BPM + Key + mode:** `librosa` (Python). Tempo via `librosa.beat.beat_track`, key via `librosa.feature.chroma_cqt` mean → Krumhansl-Schmuckler. ~1–3s per track combined.
- **Camelot wheel position:** deterministic lookup from `(key, mode)` pair. No API needed.

**Implementation note (2026-04-26):** Originally scoped as aubio (BPM) + librosa (key). aubio dropped on day 1 of implementation: its pip release hasn't been updated since 2019 and fails to compile against numpy 2.x on Python 3.14 (npy_intp ↔ long pointer-type mismatch). Switching to librosa-only is also the cleaner architecture (one less dep, no C extension to fail).

**Acceptance criteria:**
- New Track fields (optional): `bpm?: number`, `keyRoot?: string` (e.g. "A", "C#"), `keyMode?: 'major' | 'minor'`, `camelotKey?: string` (e.g. "8A", "5B").
- Computed on import for new tracks. Backfill scan for the existing library, runnable from a UI control (progress bar, cancellable, resumable).
- Persisted in `metadata-overrides.json` via the existing fingerprint-bound override path.
- Failures are non-fatal — track imports successfully even if analysis fails (e.g., DRM, corrupt file). Analysis attempt is logged so re-tries can be batched.
- No UI surface required for 4.0 Phase 0; the data is consumed by Music Man v2, Auto-DJ, and (stretch) smart playlists.

**Open decisions:**
- Run analysis **synchronously on import** (slower import) vs. **queue background analysis** (track imports immediately, BPM/key fill in later). Lean **background queue** — never block import.
- Where does the Python analysis run from? `core/` already has Python; add a new module `core/audio_analysis.py` invoked from main process via the existing Python bridge.
- Which key estimation algorithm? Krumhansl-Schmuckler is fast but ~80% accurate; libraries like Essentia are more accurate but heavier dependency. Default to librosa's built-in (KS), upgrade if accuracy is a problem.

**Risks:**
- `aubio` and `librosa` add Python dependencies — bundling impact on the packaged `.app`. Verify packaged-build size doesn't balloon.
- Python execution from Electron main is already used elsewhere in the app, so the bridge exists; only adding a new entry point.

## 3. UI Enhancement

**Direction (decided 2026-04-26 evening):** target iTunes 8 (released
September 2008). The current JakeTunes look is iTunes 6/7-era brushed
metal aluminum / aqua. iTunes 8 was the Leopard-era refinement: flat
unified gray gradient, monochrome sidebar icons, slimmer progress
bars, more matte buttons, refined typography.

**Reference image:** `docs/4.0-ui-references/itunes-8-main.png` (Jake-supplied
2026-04-26). Genius sidebar entry confirms this is iTunes 8.0 era.

**Concrete deltas iTunes 6/7 → iTunes 8:**
- Brushed metal aluminum chrome → unified gray window chrome (Leopard
  unified-toolbar style). Subtle vertical gradient, no metal texture.
- **Sidebar icons stay COLORED** in iTunes 8 (the monochrome
  conversion didn't happen until iTunes 10 / 2010). Earlier draft of
  this scope was wrong on that point — correcting.
- Glossy aqua progress / volume bars → thinner, more matte, charcoal
  fill on lighter track.
- Cover Flow gets the prominent placement when active (about half the
  main pane, dark background with reflection).
- Now-playing display is the cream/beige LCD pill — same as iTunes 7
  but slightly refined edges.

## 4. Smarter Recommendations

### 4.1 What stays the same
LLM-driven (Claude). No collaborative filtering, no embedding model.
Signal-driven prompting — feed Claude richer data, get better picks.

### 4.2 What changes

**Inputs upgraded** (depend on §2.1):
- Per-track `lastPlayedAt` → "haven't heard in a while" surfaces.
- Per-track skip count → genuine dislike signal, not just artist-level.
- Time-of-day pattern → morning vs late-night recommendations differ.

**Outputs upgraded:**
- Recommendation cards include a one-line "why this, why now" (already supported by the model — needs UI surface).
- Mixed-source recs: some from library (rediscovery), some external (new). Today the two flows are separate.

**Acceptance criteria:**
- "For You" view (or repurposed Smart Playlist) refreshes daily.
- At least three rec types: rediscovery, deep-cut from owned artist, new external album.
- Each rec carries a one-line rationale.
- Identity-based filter (§2.2) prevents external recs from suggesting owned albums.

### 4.3 Open decisions
- Is "For You" a new sidebar entry, or does it replace / extend the existing **The Music Man Picks** smart playlist?
- Refresh cadence — daily, on app open, manual? Daily on first open is the safe default.

## 5. Music Daily News

**Status:** Greenfield. Highest-risk lane in 4.0 because the data source is unsolved.

### 5.1 Gating decision: data source
The UI is the easy part. Where the news comes from determines everything else.

| Option | Pros | Cons | Risk |
|---|---|---|---|
| **RSS aggregation** (Pitchfork, Stereogum, NME, etc.) | Free, no API key, durable | Curation burden; format drift; no per-artist filtering out-of-the-box | Med |
| **News API** (NewsAPI, GNews) | Structured, per-keyword query | Cost; quality varies; music-specific filtering still needed | Med |
| **MusicBrainz + Wikipedia events** | Already integrated; release calendar data | Not really "news" — release metadata, not journalism | Low value |
| **Claude-summarized RSS** | RSS for source, Claude for filter / summarize / personalize per library | Best UX; adds Claude call cost (§2.3) | Med — and depends on §2.3 |
| **Scraping** | Most flexible | Brittle, ToS gray area, maintenance hell | High — avoid |

**Recommendation:** RSS in, Claude-summarized out, personalized against the user's library ("Beach House announced a tour — you have 4 of their albums"). Spike this before committing.

### 5.2 Acceptance criteria (assuming RSS+Claude path)
- New sidebar entry: **Daily News**.
- 5–15 cards per day, refreshed once on first app open per day.
- Each card: headline, source attribution, 2-sentence Claude summary, "you have N tracks by this artist" if relevant.
- Click-through opens external link in user's default browser.
- Cards persist across sessions for at least 7 days; older purged.
- Feature flag / settings toggle to disable entirely.

### 5.3 Spike before committing
Before writing UI:
1. Pick 4–6 RSS sources, fetch and parse them in a one-off script.
2. Run a sample through Claude with a draft summarization prompt.
3. Manually rate 20 cards: are these things Jake actually wants to see? If not, the source list is wrong; iterate before building.

## 6. Music Man v2

The existing Music Man is the strongest AI surface in the app. v2 is
**polish and consistency**, not redesign.

### 6.1 Candidate upgrades
- **Memory across sessions** — Music Man should reference past observations naturally ("last week you skipped a lot of jazz, today you're back on it").
- **Voice consistency** — currently each utterance is independent; v2 should feel like the same continuous DJ. Use prompt caching of the persona + last N utterances.
- **News integration** — Music Man can mention items from §5 conversationally ("by the way, Beach House just announced a tour").
- **Skip recovery** — when the user skips Music Man's pick, the next pick acknowledges it ("alright, not feeling that one — try this").

### 6.2 Out of scope for v2
- Voice cloning / Jake's own voice
- Multi-voice / multiple persona DJs
- Live transcription of user speech back to Music Man

## 6.5 Equalizer

A multi-band EQ in the spirit of the iTunes 4–10 era — graphical sliders,
preamp, preset menu, "Make Custom..." entry. Visual fidelity matters as
much as audio behavior here.

### 6.5.1 Implementation
- **Audio chain:** `AudioContext.createMediaElementSource(<audio>)` → 10 chained `BiquadFilterNode`s (peaking type) → `GainNode` (preamp) → `destination`.
- **Bands:** 32, 64, 125, 250, 500, 1k, 2k, 4k, 8k, 16k Hz (classic iTunes layout).
- **Range:** −12 dB to +12 dB per band; preamp same range.
- **State:** Persisted via `electron-store`. Per-track override possible (iTunes had this — track Get Info had an EQ preset field).

### 6.5.2 Presets
Match iTunes' preset list verbatim where possible (Acoustic, Bass Booster,
Classical, Dance, Deep, Electronic, Flat, Hip-Hop, Jazz, Latin, Loudness,
Lounge, Piano, Pop, R&B, Rock, Small Speakers, Spoken Word, Treble Booster,
Vocal Booster). Plus user-saved custom presets.

### 6.5.3 UI surface
- New entry under **Window** menu (or toolbar button): "Equalizer".
- Floating window (separate `BrowserWindow`) — matches iTunes behavior.
- Style spec: TBD (needs hex + reference per CLAUDE.md). Reference point:
  iTunes 7-era EQ window with metallic / aqua sliders.

### 6.5.4 Acceptance criteria
- EQ on/off toggle; "off" bypasses the entire chain (no audio coloration).
- All 21 standard presets functionally distinct (verified by ear on test tracks).
- Custom preset save / delete / rename via inline `<input>` (no `window.prompt` per CLAUDE.md).
- State persists across app restarts.
- Per-track preset override readable from Get Info modal (write path stretch goal).

### 6.5.5 Open decisions
- Floating window vs. inline panel? Floating is more iTunes-faithful but a
  new `BrowserWindow` adds IPC complexity. Lean floating; revisit if IPC
  surface area gets ugly.
- Per-track override in 4.0, or 4.x? Lean 4.x — preset persistence and the
  global EQ alone is the headline.

## 6.6 Airfoil Integration

Airfoil (Rogue Amoeba, paid third-party Mac app) extends audio routing
to Bluetooth speakers, other Macs, Airport Express units, and various
non-AirPlay targets. JakeTunes already has native AirPlay device
selection in the Toolbar; Airfoil is a **second route** for users who
have it installed.

### 6.6.1 Why Airfoil instead of (or alongside) native AirPlay
Native AirPlay only sees AirPlay-capable receivers. Airfoil bridges to:
- Bluetooth speakers (no AirPlay)
- Older AirPort Express units macOS doesn't expose cleanly
- Other Macs (via Airfoil Satellite)
- iOS/Android devices running Airfoil Satellite
- Multiple receivers simultaneously with sync
- Non-Apple smart speakers via Satellite plugins

### 6.6.2 Integration model
**Control surface:** AppleScript via `osascript` from the Electron main
process. Airfoil exposes a thorough scripting dictionary
(`tell application "Airfoil"` — list speakers, connect, disconnect,
set per-speaker volume).

**Audio source:** Airfoil captures audio per-application. We point it at
JakeTunes itself (or, if running in dev, at Electron / our specific
audio device). The user's audio chain becomes:
JakeTunes audio → system → Airfoil capture → selected Airfoil receiver(s).

**No audio re-encoding on our side.** Airfoil owns the routing.

### 6.6.3 Lifecycle
1. On app start, main process detects whether Airfoil is installed
   (`osascript -e 'exists application "Airfoil"'` or check
   `/Applications/Airfoil.app`).
2. If installed: Airfoil entry appears in the device picker alongside
   native AirPlay receivers, with a section header.
3. If not installed: feature is silently absent. No nag, no upsell.
4. On Airfoil device selection: ensure Airfoil is running (launch via
   `osascript` if not), set its source to JakeTunes, connect the
   selected speaker.
5. On switch back to native / local: disconnect from Airfoil's speakers
   gracefully. Cancel paths must mirror start paths (CLAUDE.md rule).

### 6.6.4 Risks
- **Installation gate:** ~$29 paid app. Most users won't have it. Feature must degrade silently — no broken UI when absent.
- **Airfoil version drift:** AppleScript dictionary has been stable for years but is not contractually guaranteed. Wrap all `osascript` calls in a single module so a future API change is one file's worth of repair.
- **Source-app attribution:** If JakeTunes audio is being captured by Airfoil, our own app's volume slider may behave unexpectedly (Airfoil captures pre-system-volume in some configs). Document the expected behavior; verify by ear during spike.
- **Sandboxing:** Electron main process must have permission to run `osascript`. Confirm during spike. If `electron-builder` codesign / hardened-runtime blocks AppleScript-to-other-apps, we need an entitlement
  (`com.apple.security.automation.apple-events`).

### 6.6.5 Acceptance criteria
- Device picker shows Airfoil speakers alongside native AirPlay when Airfoil is installed.
- Selecting an Airfoil speaker routes JakeTunes audio there within ~3 seconds.
- Per-speaker volume control via existing volume UI.
- Switching back to local / native AirPlay disconnects Airfoil cleanly (no zombie connections, no stuck audio).
- Feature is invisible (no menu item, no error, no greyed-out control) when Airfoil is not installed.
- Hardened-runtime entitlement in place if required, verified in the
  packaged `.app`.

### 6.6.6 Spike before committing
1. Manually run `osascript -e 'tell application "Airfoil" to get name of every speaker'` — confirm output shape.
2. From Electron main, attempt the same call and parse it.
3. Connect to one speaker, play audio from JakeTunes, confirm routing.
4. Test the no-Airfoil-installed path on a clean machine / VM.
5. Test packaged build (post-codesign) on a machine where the dev
   process worked — entitlement issues only show up in the packaged app.

### 6.6.7 Open decisions
- Show Airfoil and native AirPlay in the same picker (sectioned) vs.
  a toggle? Lean **same picker, sectioned** — matches the "second
  route" framing.
- If both Airfoil and native AirPlay can reach the same speaker (e.g., a
  HomePod), which wins? Lean **native AirPlay** for that case (lower
  latency, no third-party dependency).

## 6.7 Crossfade Songs

iTunes Preferences → Playback → "Crossfade Songs" parity. When enabled,
one track fades out as the next fades in over a configurable duration,
producing gapless playback for normal listening.

### 6.7.1 Behavior
- Toggle: off by default. When off, playback is identical to current
  behavior (one track at a time, gap on track end).
- Duration: 1–12 seconds, default 6s when enabled. Same range and default
  as iTunes 7+.
- Bypassed unconditionally when:
  - DJ Mode is active (DJ Mode runs its own Music-Man-voice transitions).
  - Repeat-one is on (would crossfade with itself).
  - There is no next track.
- Two `Howl` instances run simultaneously during the fade window. The
  current Howl's volume linearly decreases from `state.volume` to 0; the
  incoming Howl starts at 0 and increases to `state.volume`. The fade
  schedule starts at `(duration - crossfadeSeconds)`.

### 6.7.2 Settings surface
- New **Settings** modal triggered via menu **JakeTunes → Preferences…**
  (CmdOrCtrl+,). Standard macOS pattern.
- Modal layout: tabbed structure but with one section in 4.0 (Playback).
  Crossfade as a checkbox, then a slider that's only enabled when the
  checkbox is on.
- Settings persist to `app-settings.json` in `userData`. Single write per
  modal close (or per change with a debounce).

### 6.7.3 Acceptance criteria
- Toggle + slider control crossfade behavior immediately on save (no app
  restart needed).
- Settings persist across app restarts.
- DJ Mode unaffected — when DJ Mode is on, crossfade is dormant.
- No regression in playback (track end → next track start) when
  crossfade is off.
- Cancel paths (user hits Next mid-crossfade, user hits Stop mid-crossfade)
  cleanly unload both Howls and reset volume.

### 6.7.4 Gapless playback (always-on, no toggle)

Distinct from crossfade. **Crossfade overlaps + fades** two tracks. **Gapless
preloads + immediately plays** the next track on the previous one's natural
end, with no overlap and no fade — preserves the seamless boundary engineered
into albums like Pink Floyd's *The Wall* or Daft Punk's *Discovery*.

Always on (no UI surface). When the current track has ~3 seconds left,
the next track is created as a preloaded Howl in the background. On
the current track's natural end, the preloaded Howl is promoted to
`sharedHowl` (handlers attached at promote-time), bypassing the
load-and-decode latency that would otherwise insert a small audible
gap.

Bypassed when:
- Crossfade is enabled (crossfade does its own fade-with-overlap, which
  supersedes gapless).
- DJ Mode is active (DJ Mode runs its own transitions).
- Repeat=one (would preload itself).
- No next track in the queue.

### 6.7.5 Risks / edge cases
- **Two Howls = double the audio decoding load.** Should be fine for
  modern hardware but may show on older iPod-style devices or if the
  Mac is otherwise busy. Run a single fade animation frame loop — don't
  spawn two parallel timers.
- **User's volume control during crossfade.** If the user adjusts volume
  while a fade is in progress, both Howls' target volumes need to track
  the new max. Easiest: at every animation frame, read `state.volume`
  and compute current/incoming as fractions of it.
- **Seek during crossfade.** If the user seeks the current track back
  into pre-fade territory, the crossfade should abort cleanly (cancel
  the secondary Howl, restore primary volume to `state.volume`).
- **Tracks shorter than the crossfade duration.** A 5-second interlude
  with crossfadeSeconds=12 doesn't have room. Fall back to "no crossfade,
  natural end" for tracks where `duration < crossfadeSeconds + 1`.

## 9. Sequencing

Rough order; each phase gates the next.

1. **Phase 0 — Foundation** (§2): signal layer, identity dedup, API posture. ~1–2 weeks.
2. **Phase 1a — Daily News spike** (§5.3): prove the RSS+Claude pipeline.
3. **Phase 1b — Airfoil spike** (§6.6.6): prove `osascript` control + entitlements survive codesign. Parallel with 1a.
4. **Phase 2 — Recommendations upgrade** (§4): builds on Phase 0 signals.
5. **Phase 3 — EQ** (§6.5): self-contained; can run in parallel with Phase 2.
6. **Phase 4 — Daily News UI** (§5.2): builds on Phase 1a spike.
7. **Phase 5 — Airfoil integration** (§6.6): builds on Phase 1b spike.
8. **Phase 6 — Music Man v2** (§6): builds on §5 + §4 already shipped.
9. **Phase 7 — UI enhancement** (§3): runs in parallel with Phase 2–6 once §3 has specs.

UI enhancement is sequenced last in this list only because it's
under-specified today. Once §3 has hex values and references, it
parallelizes with everything.

Spikes (Phase 1a, 1b) are intentionally early — both have unknowns
that could force a re-scope if they fail. Better to find out in
days than in a finished UI.

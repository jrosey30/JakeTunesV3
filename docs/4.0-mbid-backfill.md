# 4.0 Brief — MusicBrainz Release Group ID storage + library backfill

**Status:** Draft, 2026-04-26
**Parent:** `docs/4.0-scope.md` §2.2 (deferred from Phase 0)
**Owner:** Jake

---

## 1. Goal

Add a stable, language-agnostic identifier (MusicBrainz Release Group ID, "RGID") to every album in the library so that recommendation dedup, Cynthia's investigations, and any future album-level identity work can gate on identity instead of text comparison.

This brief covers the foundation only. Consumers (rec dedup, Cynthia, future "have I heard this album before" surfaces) attach in their own briefs.

## 2. Why this exists

**Phase 0 §2.2** intentionally shipped a smarter text-based matcher and noted that real identity-based dedup requires MBID storage. The smarter matcher catches more cases ("Pt." ↔ "Part", parenthetical suffixes, articles) but is still text-only — it will eventually false-positive on genuinely different albums with similar titles, and it can't catch the inverse failure (model recommends "Beach House — Once Twice Melody (Part 1)" and we don't dedupe it against the user's "Beach House — Once Twice Melody, Pt. 1" because edit-distance is too high).

RGID is the only durable fix. MusicBrainz is open, well-curated, and already integrated in five places in `main/index.ts`. The cost is a one-time backfill scan plus per-import lookup going forward.

## 3. Scope

### In scope
- New optional field `releaseGroupId?: string` on the Track interface.
- Backfill scan that runs MusicBrainz lookups for every (artist, album) pair in the library and writes `releaseGroupId` per track.
- A polite throttler shared with all MB code paths so future MB-using features can use it without re-implementing.
- Failure markers so unmatched tracks aren't re-queried every scan.
- Triggered from a new "Music Man → Organize Library" action, with a progress UI.
- Persistence via the existing `metadata-overrides.json` v2 entry format (no schema migration).

### Out of scope (separate briefs)
- Recommendation dedup using RGID — separate brief, depends on this one landing.
- Cynthia migration to RGID-keyed lookup — separate brief.
- Per-track recording IDs (more granular than release group; not needed for dedup).
- Auto-backfill on app start (we'll require explicit user kickoff for Phase 1; revisit after we see scan timing in real libraries).

## 4. Data model

Add to `Track` in `src/renderer/types.ts`:

```typescript
// MusicBrainz Release Group ID. Same value for every track on the same
// album across all editions (original, remaster, deluxe, etc.). Empty
// string is the "tried, no match" sentinel — see §6.4.
releaseGroupId?: string

// Epoch ms of the last MusicBrainz lookup attempt. Used to gate retries
// for tracks that previously came back empty.
mbLookupAt?: number
```

Persisted as override fields:
- `releaseGroupId` (string, value-as-string in the override JSON, applied directly)
- `mbLookupAt` (numeric, coerced via the App.tsx `NUMERIC_OVERRIDE_FIELDS` set added in §2.1)

No schema version bump. No migration. Existing tracks lacking the fields are simply candidates for the next backfill scan.

## 5. MusicBrainz throttler

MB allows ~1 request/sec for anonymous use; aggressive callers get a 503 with a `Retry-After` header. Build a single shared queue:

```typescript
// In main/index.ts, near the existing MB callers (~line 350)
class MusicBrainzQueue {
  // serialized fetch with 1100ms minimum gap (jitter to avoid lockstep
  // on retries), 5xx → exponential backoff up to 60s, 4xx → fail-fast.
  // User-Agent header MUST identify the app per MB's terms — bake in
  // "JakeTunes/3.x.x ( jakerosenbaum30@gmail.com )".
}
```

All five existing MB callers (`searchMusicBrainz`, the CD-import lookup at ~line 3920, the Cynthia tool at ~line 2960, plus the two release-group-fetching paths) get migrated to use this queue in the same change. **This is a small refactor that should land alongside the backfill, not after.** Otherwise the backfill will fight existing inline calls for rate limit headroom.

Not strictly required by MB but wise: cache lookup results to disk in `mb-cache.json` keyed by `(artist|album)`. Two motivations: (1) backfill resumability — a crash mid-scan doesn't lose work; (2) future MB callers (Cynthia, recs) get instant hits for already-looked-up albums.

## 6. Backfill scan

### 6.1 Trigger
New entry under **Music Man → Organize Library** view: "Add MusicBrainz IDs". Click → opens a confirmation modal explaining duration estimate ("about N minutes for ~M albums") → starts.

Not auto-triggered on app start. Reasoning: a user with 10k albums would face a ~3-hour delay on first launch with no expectation set; better to make it explicit.

### 6.2 Unit of work
Group tracks by `(albumArtist || artist, album)`. Look up the album once; write the resulting RGID to every track in that group. Reduces lookup count from ~tracks-in-library to ~albums-in-library (roughly 10×).

### 6.3 Match strategy
For each (artist, album):
1. Query `release-group` endpoint with `artist:"<a>" AND releasegroup:"<t>"`, limit 5.
2. Filter results: must match artist (case-insensitive, accent-folded) AND title (Levenshtein ≤ 2 OR exact normalized match using the §2.2 `normForOwnership` matcher).
3. If multiple candidates pass: prefer earliest `first-release-date`, then most "popular" (`tag-count` highest). Document the tie-breaker.
4. If zero candidates pass: write the empty-string sentinel + `mbLookupAt`.

Reuse Cynthia's pattern from `main/index.ts:~2960` for selecting among multiple MB candidates — that code already handles the disambiguation logic well (e.g., spotting wrong-artist-same-name).

### 6.4 Failure markers
A track that comes back empty gets `releaseGroupId: ''` (empty string, not absent) and `mbLookupAt: <epoch>`. Re-scan logic:
- Skip tracks where `mbLookupAt` is set AND the timestamp is < 30 days old.
- Re-attempt tracks whose timestamp is older (MB updates entries; old failures sometimes succeed later).
- Always re-attempt tracks where `releaseGroupId` is undefined (never tried).

### 6.5 Resumability
Cache writes after every batch of 10 lookups. If the user closes the app or the scan crashes, restart from the next un-attempted track. The `mbLookupAt` sentinel guarantees no duplicate work.

### 6.6 Progress UI
- Modal with progress bar `<n> / <total albums>`.
- Per-step status: `Looking up "Artist — Album"...`
- Found count and not-found count visible during scan.
- Cancel button — sets a flag the loop checks every iteration; cleanly exits within one MB request (~1.1s).
- Final summary: "Scanned N albums. Found IDs for M, no match for K. Try again later for the K unmatched."

## 7. Future hooks for consumers

This brief doesn't change any feature today. But the consumers it enables should know what to expect:

- **Recommendation dedup (§2.2 follow-up):** primary key is RGID where both sides have one. When the rec from Claude doesn't carry an RGID (most cases), we look it up server-side via the same throttled queue; if MB has no match, fall back to the §2.2 smart-text matcher.
- **Cynthia investigations:** her existing musicbrainz_album_lookup tool can be extended to use the cached RGID first, hitting MB only when not cached. Significant latency win.
- **"You've already heard this" surfaces:** any future "you have an album like this" UX (e.g., import-time duplicate detection) gets a free identity primary key.

## 8. Acceptance criteria

- New `releaseGroupId?: string` and `mbLookupAt?: number` fields on Track, both persisted via existing override path with fingerprint binding.
- One shared `MusicBrainzQueue` in `main/index.ts`. All existing MB callers migrated to use it.
- "Add MusicBrainz IDs" entry in the Music Man Organize Library view.
- Modal with progress, cancel, resume-from-where-it-left-off semantics.
- 30-day failure cooldown — unmatched tracks aren't re-queried within 30 days.
- `mb-cache.json` at userData with `(artist|album)` → RGID mappings, persisted between sessions.
- After a scan, every track in the library has either a non-empty `releaseGroupId` OR a `mbLookupAt` timestamp.
- Polite User-Agent header on every MB request per MB's terms.
- The smart-text matcher in §2.2 is **not** removed. It becomes the fallback when RGID isn't present on either side of a comparison.

## 9. Risks and edge cases

- **Multi-disc compilations / various-artists releases.** "Various Artists — Now That's What I Call Music 50" is a MB release group. Track-level artists differ from the album artist. Matching strategy must use `albumArtist` not `artist` when grouping. Tracks where `albumArtist` is blank fall back to `artist`.
- **Live albums vs studio albums of the same name.** "Genesis — Genesis" (1983 studio) and "Genesis — Genesis Live" (1973) — different release groups, must not collapse. Title match must be exact-or-very-close.
- **Self-titled albums.** "Beach House — Beach House" — short titles increase false-match risk. Tighten the title-distance threshold for titles ≤ 5 characters.
- **Bandcamp / vinyl-only / niche releases.** MB has no entry. These tracks will permanently carry the empty-string sentinel after the 30-day retry. That's correct behavior; downstream consumers fall back to text matching.
- **Albums released under different artist names.** "Iron and Wine — The Shepherd's Dog" vs "Iron & Wine — The Shepherd's Dog" — handled by the existing artist-normalization in §2.2 (the smart matcher is reused as the title/artist comparator inside the candidate filter).
- **MB rate-limit drift / outages.** Backfill must survive 503s gracefully (back off, resume), and the user-visible UI must NOT report "failed" — the scan pauses and continues. Only a true cancel or a 4xx aborts.
- **Cancel-then-resume identity.** If the user cancels mid-scan and starts again, the second run must skip already-looked-up albums (via the cache) and not re-bill the rate limit budget for them.

## 10. Open decisions

1. **`mbLookupAt` cooldown duration** — 30 days is a guess. Consider 7d / 30d / 90d. Lean 30d for a balance of fresh re-tries vs. not-spamming-MB. (Decision point: Jake.)
2. **Recommendation-side lookup** — when Claude returns recs without RGIDs, do we (a) look them up server-side before dedup, (b) skip MBID dedup for recs and rely on smart-text matcher only, or (c) ship without and revisit? Lean (a); ~10 MB calls per recs request is well under rate limit. (Decision point: Jake.)
3. **Auto-trigger ever?** Right now: user clicks. Alternative: silently scan in the background after first launch with no UI noise. Risk is the user gets traffic they didn't ask for; benefit is they don't have to discover the feature. Lean stay-explicit for Phase 1; revisit after seeing real-world scan times.
4. **Cynthia integration timing** — migrate her MB calls to use the shared queue + cache *in this brief*, or defer to a Cynthia-specific brief? Lean migrate-now (avoids rate-limit collisions during a backfill while Cynthia is also being used).
5. **What if MB is unreachable?** Scan can't run. Show a clear "MusicBrainz is unreachable, try again later" rather than just silently doing nothing. Acceptance criteria stricter than the rest because this is a primary path.

## 11. Sequencing

1. **MB queue + cache** (~half day). Refactors all five existing callers. No user-visible change. Type-check + manual smoke of CD import (the most-exercised MB path) after.
2. **Track type fields + override coercion** (~30 min). Mirrors §2.1's lastPlayedAt/skipCount additions; same patterns.
3. **Backfill engine + IPC handlers** (~half day). Stateful loop, cancel, resume, progress events.
4. **UI: modal in Organize Library** (~half day). Progress bar, cancel, summary.
5. **Spec the consumer integration** (separate brief). RGID-aware rec dedup composes with §2.2 smart-text matcher.

Total: ~2 focused days for steps 1–4. Step 5 is separate work.

## 12. Out-of-this-brief signals worth noting

- Once RGID is in, Cynthia's investigations get cheaper *and* more accurate (no more "wrong artist with same name" disambiguation calls).
- Music Daily News (§5) can use RGID to detect when a news item is about an album you already own ("Beach House announced a tour — you have 4 of their albums" gets a precise count).
- §2.4 (audio analysis) is independent of this work; both can run in parallel without coupling.

## Appendix A — Sample MB query

```http
GET https://musicbrainz.org/ws/2/release-group/?query=artist:%22Beach%20House%22%20AND%20releasegroup:%22Once%20Twice%20Melody%22&fmt=json&limit=5

User-Agent: JakeTunes/4.0.0 ( jakerosenbaum30@gmail.com )
```

Response (truncated):

```json
{
  "release-groups": [{
    "id": "5a4f0a1c-...",
    "title": "Once Twice Melody",
    "primary-type": "Album",
    "first-release-date": "2022-02-18",
    "artist-credit": [{ "name": "Beach House" }],
    "tags": [{ "count": 12, "name": "dream pop" }]
  }]
}
```

The `id` is the RGID we persist.

## Appendix B — Cache file shape

`{userData}/mb-cache.json`:

```json
{
  "beach house|once twice melody": {
    "rgid": "5a4f0a1c-...",
    "ts": 1714150823456
  },
  "various artists|now that's what i call music 50": {
    "rgid": "",
    "ts": 1714150823999
  }
}
```

Key normalization uses the §2.2 `normForOwnership` function so cache hits survive minor case/punctuation differences in the lookup query.

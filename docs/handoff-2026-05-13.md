## What I tried tonight and FAILED to do

Jake reported: *"The syncing is not seamless. Today at work I was
using my screen share to play music (homemini has JakeTunes) and the
homemini version had issues getting my newest music in the library."*

I tried to address today's pain by getting the laptop's 68GB music
tree to the NAS so both machines could read from it (Phase 1 band-aid).
**I failed.** Here's the honest sequence:

### What I assumed (wrongly)

1. Jake said "Plex's `/volume1/Music/JakeTunesLibrary`" — I treated
   this as "files are already on the NAS via Plex's storage."
2. The NAS exposes an SMB share named `Music2` (confirmed via Finder
   dialog: shares are `jakerosenbaumnas`, `Macintosh HD`, `Music2`,
   `Public Folder`).
3. I assumed `Music2` SMB share mapped to the same NAS filesystem
   location Plex sees.

### What's actually true

The SMB share named `Music2` does NOT map to `/volume1/Music/`
(where Plex sees the music). It maps to some other NAS folder
(`/volume1/Music2/`?) that's mostly empty. Evidence:

- `ls /Volumes/Music2 | head -10` showed `JakeTunesLibrary` (one entry)
- `ls /Volumes/Music2/JakeTunesLibrary/` → "No such file or directory"
- `touch /Volumes/Music2/JakeTunesLibrary/.write-test` → "No such file or directory"

So the SMB `Music2` share contains a `JakeTunesLibrary` shell with
no contents — a hollow directory entry that errors on enter.

### What that means for rsync

Every rsync attempt failed:

1. First attempt (`rsync -avh --ignore-existing ~/Music2/JakeTunesLibrary/ /Volumes/Music2/JakeTunesLibrary/`)
   → `mkdir failed: File exists (17)` on the top dir. macOS SMB
   returns EEXIST on chmod of an existing dir; rsync chokes.
2. Switched flags to `-rltvh` (dropping perm/owner preservation that
   confuses SMB) → same EEXIST error. Flags didn't fix it because the
   chmod-on-existing-dir issue happens whether you preserve perms or not
   (this part of my reasoning was wrong).
3. Descended one level (`.../iPod_Control/.../iPod_Control/`)
   → `mkdir failed: No such file or directory (2)`. New error: the
   parent path (`JakeTunesLibrary` on the NAS share) is hollow.
4. Tried `touch` on the same NAS path → also ENOENT.

### Root cause (correct this time)

**The NAS has no SMB share exposing `/volume1/Music/JakeTunesLibrary`
(where Plex scans).** The `Music2` SMB share Jake saw and mounted is
a different folder, mostly empty. So none of the rsync attempts had
any chance of working — they were writing to the wrong NAS folder.

We don't have a writable network path from the laptop to where Plex's
music lives. The Synology has the data; just no SMB share exposes it.

### What needs to happen next

**Option A — Create the SMB share** (Synology Web UI, ~5 minutes):
- Browser to `http://192.168.1.50:5000` → Control Panel → Shared
  Folder → Create
- Name it `Music` (or whatever), filesystem path `/volume1/Music/`
- Permissions: Jake's user, read/write
- Then `open 'smb://192.168.1.50/Music'` → mount → rsync targets
  `/Volumes/Music/JakeTunesLibrary/`. Should work cleanly because
  the share would be properly exposed.

**Option B — Fix SSH and rsync over SSH directly:**
- Yesterday's `ssh jakerosenbaumnas@192.168.1.50` returned "Broken
  pipe." Could be a stale known_hosts entry or a one-off network
  blip; haven't retried.
- If SSH works: `rsync -avh ~/Music2/JakeTunesLibrary/ jakerosenbaumnas@192.168.1.50:/volume1/Music/JakeTunesLibrary/`
- No SMB involvement → no EEXIST quirks.

**Option C — Synology File Station drag-drop / Synology Drive:**
- Slow GUI option. Reliable but tedious for 68GB.

**My recommendation for whoever picks this up:** start with Option B.
SSH would have failed by now if it were permanently broken; one
retry is cheap. If it works, the whole rsync drama is over in 10
minutes. If it doesn't, fall back to A.

### What I should have done differently

1. **Verified the SMB share's actual filesystem mapping before any
   rsync attempt.** I should have run `smbutil view -A` or checked
   Synology's share config first.
2. **Tested write to the share root, not the deep path.** A single
   `touch /Volumes/Music2/.test` would have told me whether the
   share was even writable AT ALL. Three iterations of flag-tweaking
   on rsync was wasted motion.
3. **Stopped sooner.** When the second rsync error was different from
   the first (EEXIST → ENOENT), that was a signal something
   fundamentally weird was happening with the path, not just a flag
   issue. I kept tweaking rsync instead of stepping back.

Jake was extremely patient. He shouldn't have had to be.
